## 1. Executive Summary

The `scaffold-plugin.ts` TypeScript script (generated by `generate_scaffold_script()` in Rust and embedded in the starter plugin) creates new plugin directory structures under `.ai/` but does not register them for discovery or activation. This spec adds two post-creation steps to the generated script: (1) append the new plugin entry to `.ai/.claude-plugin/marketplace.json` and (2) auto-enable the plugin in `.claude/settings.json` `enabledPlugins`. A snapshot test of the generated TypeScript output is added alongside updates to existing unit, e2e, and BDD test layers.

## 2. Context and Motivation

### 2.1 Current State

The starter plugin's scaffold script is a TypeScript file embedded as a Rust string literal in `generate_scaffold_script()` at [`crates/libaipm/src/workspace_init/mod.rs:285-325`](../crates/libaipm/src/workspace_init/mod.rs). When a user creates a new plugin via the scaffold-plugin skill:

```bash
node --experimental-strip-types .ai/starter-aipm-plugin/scripts/scaffold-plugin.ts my-new-plugin
```

The script creates ([research §1](../research/docs/2026-03-20-scaffold-plugin-ts-missing-features.md)):
1. `.ai/my-new-plugin/.claude-plugin/plugin.json`
2. `.ai/my-new-plugin/aipm.toml`
3. `.ai/my-new-plugin/skills/my-new-plugin/SKILL.md`
4. `.ai/my-new-plugin/agents/` (empty)
5. `.ai/my-new-plugin/hooks/` (empty)

It then prints a success message and exits.

Meanwhile, the Rust-side `aipm init` command correctly handles both marketplace registration (`generate_marketplace_json()` at `mod.rs:371-403`) and autoenable (`adaptors/claude.rs:28-42`) for the **starter** plugin. The TypeScript script simply does not replicate these two steps for **newly scaffolded** plugins.

### 2.2 The Problem

| Problem | Impact |
|---------|--------|
| New plugins are not added to `marketplace.json` | Claude Code cannot discover the plugin through its marketplace scanning — the plugin is invisible |
| New plugins are not added to `enabledPlugins` in `settings.json` | Even if discovered, the plugin is not activated — users must manually edit `settings.json` |
| No snapshot test for the TypeScript output | The generated TypeScript is a Rust string literal that could silently regress without detection; content assertions are limited to 3 keyword checks (`mkdirSync`, `writeFileSync`, `experimental-strip-types`) |

## 4. Proposed Solution (High-Level Design)

### 4.1 Architectural Pattern

**Template Generation with Embedded Scripts** — follows the existing pattern established by `generate_scaffold_script()` and other `generate_*()` functions. The TypeScript source is a Rust string literal that gets written to disk during `scaffold_marketplace()`. The change is purely additive: two new TypeScript functions are appended to the generated script.

## 5. Detailed Design

### 5.1 Updated TypeScript: marketplace.json Registration

After writing the plugin files (existing behavior), the script reads `.ai/.claude-plugin/marketplace.json`, appends the new plugin entry, and writes it back.

**New TypeScript logic (appended after `plugin.json` write):**

```typescript
import { mkdirSync, writeFileSync, readFileSync, existsSync } from "fs";
import { join } from "path";

// ... existing plugin directory creation ...

// Register in marketplace.json
const marketplacePath = join(aiDir, ".claude-plugin", "marketplace.json");
type MarketplaceData = {
  name: string;
  owner: { name: string };
  metadata: { description: string };
  plugins: Array<{ name: string; source: string; description: string }>;
};
let marketplace: MarketplaceData;

if (existsSync(marketplacePath)) {
  marketplace = JSON.parse(readFileSync(marketplacePath, "utf-8")) as MarketplaceData;
} else {
  marketplace = {
    name: "local-repo-plugins",
    owner: { name: "local" },
    metadata: { description: "Local plugins for this repository" },
    plugins: []
  };
  mkdirSync(join(aiDir, ".claude-plugin"), { recursive: true });
}

if (!marketplace.plugins.some((p: { name: string }) => p.name === name)) {
  marketplace.plugins.push({
    name,
    source: `./${name}`,
    description: `TODO: describe ${name}`
  });
  writeFileSync(marketplacePath, JSON.stringify(marketplace, null, 2) + "\n");
}
```

**Conventions followed** ([research §2](../research/docs/2026-03-20-scaffold-plugin-ts-missing-features.md)):
- `source` is the directory name prefixed with `./` (relative to `.ai/`)
- `description` matches the `aipm.toml` package description placeholder
- The marketplace top-level `name` is `"local-repo-plugins"`
- Duplicate detection via `plugins.some(p => p.name === name)`

### 5.2 Updated TypeScript: settings.json Autoenable

After updating marketplace.json, the script reads `.claude/settings.json` and adds the plugin to `enabledPlugins`.

**New TypeScript logic:**

```typescript
// Auto-enable in .claude/settings.json
const settingsPath = join(process.cwd(), ".claude", "settings.json");
type SettingsData = {
  enabledPlugins?: Record<string, boolean>;
  [key: string]: unknown;
};

let settings: SettingsData;
if (existsSync(settingsPath)) {
  settings = JSON.parse(readFileSync(settingsPath, "utf-8")) as SettingsData;
} else {
  mkdirSync(join(process.cwd(), ".claude"), { recursive: true });
  settings = {};
}

const pluginKey = `${name}@local-repo-plugins`;
if (!settings.enabledPlugins) {
  settings.enabledPlugins = {};
}
if (!(pluginKey in settings.enabledPlugins)) {
  settings.enabledPlugins[pluginKey] = true;
  writeFileSync(settingsPath, JSON.stringify(settings, null, 2) + "\n");
}
```

**Key format** ([research §3](../research/docs/2026-03-20-scaffold-plugin-ts-missing-features.md)):
- `<plugin-name>@local-repo-plugins` — matches the pattern used in `adaptors/claude.rs:40,95`
- Existing settings content is preserved (read-modify-write, not overwrite)
- The `enabledPlugins` object is created if missing

### 5.3 Updated `generate_scaffold_script()` in Rust

The function at `mod.rs:285-325` is modified to embed the updated TypeScript. The key changes to the Rust string literal:

1. **Import change**: Add `readFileSync` to the `fs` import
2. **New section**: marketplace.json read/update/write after plugin.json creation
3. **New section**: settings.json read/update/write after marketplace.json update
4. **Updated success message**: Include registration status

The Rust function signature and return type remain unchanged — it still returns `String`.

### 5.4 Error Handling in TypeScript

The script uses a **warn-and-continue** strategy for registration failures, distinct from the **exit-on-error** strategy for directory creation:

| Operation | Error Strategy | Rationale |
|-----------|---------------|-----------|
| Plugin directory exists | `process.exit(1)` (existing) | Core failure — nothing to create |
| Missing plugin name arg | `process.exit(1)` (existing) | Can't proceed without a name |
| marketplace.json parse failure | `process.stderr.write(warning)`, skip | Plugin files are created; registration is best-effort |
| settings.json parse failure | `process.stderr.write(warning)`, skip | Plugin files are created; user can manually enable |
| marketplace.json write failure | `process.stderr.write(warning)` | Non-fatal — files exist, user can manually register |
| settings.json write failure | `process.stderr.write(warning)` | Non-fatal — files exist, user can manually enable |

This ensures the plugin directory is always created even if the registration files are corrupted or locked.

### 5.5 Edge Cases

| Edge Case | Behavior |
|-----------|----------|
| `marketplace.json` doesn't exist | Create it with default structure (name, owner, metadata, empty plugins array), then append |
| `settings.json` doesn't exist | Create `.claude/` directory and new settings with just `enabledPlugins` |
| `enabledPlugins` key missing from `settings.json` | Create the key as an empty object, then add the entry |
| Plugin already in `marketplace.json` | Skip append (no duplicates) |
| Plugin already in `enabledPlugins` | Skip add (no duplicates) |
| JSON parse error in either file | Warn to stderr, skip that registration step, continue |

## 6. Alternatives Considered

| Option | Pros | Cons | Reason for Rejection |
|--------|------|------|---------------------|
| **Separate registration script** (`register-plugin.ts`) | Clean separation of concerns; scaffold and register are independent | Two scripts to maintain; users must remember to run both; breaks the "one command" UX | The scaffold skill should be a single-action experience |
| **Registration in the SKILL.md instructions** (tell AI to manually edit JSON) | No code change needed | Unreliable — AI may not follow instructions perfectly; JSON editing is error-prone | Deterministic script execution is more reliable than AI-directed JSON editing |
| **Rust-side registration via `aipm register`** command | Type-safe, tested in Rust, no TypeScript complexity | Requires implementing a new `aipm` subcommand; users need both `aipm` and `node` | Over-engineering for a starter template; the TypeScript approach matches the existing pattern |
| **Read marketplace name from marketplace.json** instead of hardcoding | More robust if marketplace name changes | Adds complexity; the marketplace name `local-repo-plugins` is set by `aipm init` and never changes | Hardcoding matches the existing convention in both `generate_marketplace_json()` and `adaptors/claude.rs` |

## 7. Cross-Cutting Concerns

### 7.1 Platform Considerations

- **Windows**: `readFileSync`/`writeFileSync` use forward-slash paths from `join()`. Node.js handles path normalization cross-platform. No special Windows handling needed.
- **Node.js version**: Requires Node.js >= 22.6.0 with `--experimental-strip-types` (unchanged from current requirement, [better-default-plugin spec §7.3](../specs/2026-03-20-better-default-plugin.md)).
- **Encoding**: All JSON files are UTF-8. `readFileSync` with `"utf-8"` encoding is used consistently.

### 7.2 Data Integrity

- **Read-modify-write atomicity**: JSON files are read, modified in memory, and written back. No file locking. If two scaffold operations run concurrently, the last write wins. This is acceptable for a developer tool where concurrent scaffolding is extremely unlikely.
- **Pretty-printing**: `JSON.stringify(data, null, 2) + "\n"` matches the format used by `serde_json::to_string_pretty` in the Rust Claude adaptor (`claude.rs:99`), ensuring consistent formatting.

### 8.2 Test Plan

#### 8.2.1 Snapshot Test (New — `workspace_init/mod.rs`)

Add the `insta` crate as a dev-dependency to `crates/libaipm/Cargo.toml`. Create a snapshot test that captures the entire `generate_scaffold_script()` output:

```rust
#[test]
fn scaffold_script_snapshot() {
    let content = generate_scaffold_script();
    insta::assert_snapshot!(content);
}
```

This produces a `.snap` file at `crates/libaipm/src/workspace_init/snapshots/libaipm__workspace_init__tests__scaffold_script_snapshot.snap` containing the full TypeScript source. Any change to the generated script requires an explicit `cargo insta review` approval, preventing silent regressions.

**Why snapshot testing?** The TypeScript is a ~60-line program embedded as a Rust string literal. Keyword-based assertions (`contains("mkdirSync")`) catch gross omissions but miss subtle bugs like incorrect JSON key names, missing `readFileSync` calls, or wrong marketplace name strings. A snapshot captures the exact output and forces review of any change.

## 9. Open Questions / Unresolved Issues

- [ ] **`insta` crate version**: Which version of `insta` should be added to `Cargo.toml`? The latest stable should be fine as a dev-dependency. Verify it works with the workspace lint configuration (the `#[allow]` that `insta` macros may emit internally — derive macros are permitted per `CLAUDE.md`).
- [ ] **Node.js in CI**: Do the GitHub Actions CI runners have Node.js >= 22.6.0? If not, the e2e scaffold-script tests must be gated with `#[ignore]` or a runtime check. The BDD scaffold scenarios would need the same gating.
- [ ] **Concurrent scaffolding**: The read-modify-write pattern for JSON files is not atomic. If two scaffold operations run simultaneously, the second write could overwrite the first's changes. This is extremely unlikely for a developer tool but worth documenting. No fix proposed — accepted risk.
- [ ] **Marketplace name coupling**: The TypeScript hardcodes `"local-repo-plugins"` as the marketplace name. If `generate_marketplace_json()` ever changes this name, the scaffold script will produce inconsistent keys. Should we add a unit test that cross-checks the two? **Recommendation**: Yes — add a test that extracts the marketplace name from `generate_marketplace_json()` and asserts the scaffold script contains the same string.
