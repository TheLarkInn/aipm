# Research: `aipm lint` Command — Issue #110

## Research Question

What existing codebase patterns, architecture, and conventions inform the design of `aipm lint`? Specifically: (1) the adapter architecture from `aipm migrate`, (2) the plugin/marketplace structure and conventions that lint rules would validate, and (3) any existing validation or rule-based patterns.

## Summary

The codebase has a well-established **trait-based detector pattern** in the migrate pipeline that directly maps to a lint rule architecture. The `Detector` trait (`detect(source_dir, fs) -> Vec<Artifact>`) provides the template — a `LintRule` trait would follow the same shape but return diagnostics instead of artifacts. The filesystem abstraction (`Fs` trait) enables testable rules without touching the real filesystem. Manifest validation already aggregates multiple errors via `Error::Multiple`. The BDD feature file at `tests/features/guardrails/quality.feature` specifies the expected behavior of `aipm-pack lint` including severity levels, `--fix` mode, and publish gating. The `ArtifactKind` enum and marketplace conventions define what the lint rules need to validate.

## Detailed Findings

### 1. Migrate Adapter (Detector) Architecture

The migrate pipeline uses a **strategy pattern** with trait objects for scanning different artifact types from `.claude/` directories.

#### The Detector Trait

[`crates/libaipm/src/migrate/detector.rs:13-20`](https://github.com/TheLarkInn/aipm/blob/b034f7a3c3326ea746e8afd8bee63a7170899ca4/crates/libaipm/src/migrate/detector.rs#L13-L20)

```rust
pub trait Detector {
    fn name(&self) -> &'static str;
    fn detect(&self, source_dir: &Path, fs: &dyn Fs) -> Result<Vec<Artifact>, Error>;
}
```

- Takes a `source_dir` path and an `&dyn Fs` filesystem abstraction
- Returns zero or more `Artifact` values (the findings)
- Each detector is a zero-sized struct implementing this trait

#### Detector Factory (Registration)

[`crates/libaipm/src/migrate/detector.rs:23-32`](https://github.com/TheLarkInn/aipm/blob/b034f7a3c3326ea746e8afd8bee63a7170899ca4/crates/libaipm/src/migrate/detector.rs#L23-L32)

A factory function `claude_detectors()` returns `Vec<Box<dyn Detector>>` containing all six detectors. Source-type selection happens in `mod.rs:264-267` — a `match` on the source string dispatches to `claude_detectors()` for `".claude"` and returns `Error::UnsupportedSource` for anything else.

#### Two-Phase Pipeline

1. **Detection phase** (read-only, parallel-safe via `rayon::par_iter()`): produces `Artifact` values
2. **Emission phase** (writes): consumes artifacts to create plugin directories

The lint command would only need a detection-like phase (read-only scan producing diagnostics).

#### Source-Type Extension Point

[`crates/libaipm/src/migrate/mod.rs:264-267`](https://github.com/TheLarkInn/aipm/blob/b034f7a3c3326ea746e8afd8bee63a7170899ca4/crates/libaipm/src/migrate/mod.rs#L264-L267)

The `match source` dispatches to `claude_detectors()` for `".claude"`. Currently returns `Error::UnsupportedSource` for other sources. Issue #110 requires the same adapter pattern so lint can run against `.github/copilot`, `.opencode`, etc.

#### Parallel Execution

Both detection and emission use `rayon::par_iter()` in recursive mode ([`mod.rs:331`](https://github.com/TheLarkInn/aipm/blob/b034f7a3c3326ea746e8afd8bee63a7170899ca4/crates/libaipm/src/migrate/mod.rs#L331) and [`mod.rs:401`](https://github.com/TheLarkInn/aipm/blob/b034f7a3c3326ea746e8afd8bee63a7170899ca4/crates/libaipm/src/migrate/mod.rs#L401)).

#### Filesystem Abstraction

[`crates/libaipm/src/fs.rs:25`](https://github.com/TheLarkInn/aipm/blob/b034f7a3c3326ea746e8afd8bee63a7170899ca4/crates/libaipm/src/fs.rs#L25)

The `Fs` trait (`Send + Sync`) provides `exists`, `create_dir_all`, `write_file`, `read_to_string`, `read_dir`. Production uses `Real`; tests use mocks. All detectors accept `&dyn Fs`.

### 2. Plugin/Marketplace Structure and Conventions

#### `.ai/` Marketplace Layout

```
.ai/
├── .gitignore                           # Managed markers for symlinked registry plugins
├── .claude-plugin/
│   └── marketplace.json                 # Registry of all plugins
├── <plugin-name>/                       # Each plugin is a subdirectory
│   ├── .claude-plugin/plugin.json       # Claude Code plugin descriptor
│   ├── aipm.toml                        # (optional) aipm manifest
│   ├── skills/<name>/SKILL.md           # Skill definitions (directory-based)
│   ├── commands/<name>.md               # Command definitions (single file, legacy)
│   ├── agents/<name>.md                 # Agent definitions (single file)
│   ├── hooks/hooks.json                 # Hook definitions (JSON)
│   ├── .mcp.json                        # MCP server config (JSON)
│   ├── scripts/                         # Utility scripts
│   └── README.md, LICENSE               # Documentation
```

[`.ai/.claude-plugin/marketplace.json`](https://github.com/TheLarkInn/aipm/blob/b034f7a3c3326ea746e8afd8bee63a7170899ca4/.ai/.claude-plugin/marketplace.json)

Contains `name`, `owner.name`, `metadata.description`, and `plugins[]` array where each entry has `name`, `source` (relative path), and `description`.

### 3. Existing Validation and Rule-Based Patterns

#### Manifest Validation (Error Accumulation)

[`crates/libaipm/src/manifest/validate.rs:76-192`](https://github.com/TheLarkInn/aipm/blob/b034f7a3c3326ea746e8afd8bee63a7170899ca4/crates/libaipm/src/manifest/validate.rs#L76-L192)

The `validate()` function collects errors into a `Vec<Error>` and dispatches to three sub-validators:

1. **`validate_package()`** (line 113): checks name non-empty + valid pattern, version semver, plugin type
2. **`validate_dependencies()`** (line 140): checks each dep version string is parseable
3. **`validate_component_paths()`** (line 167): checks every declared component path exists on disk

Aggregation pattern (line 103-110): if 1 error, return it directly; if multiple, wrap in `Error::Multiple(Vec<Self>)`.

Name validation at line 14 enforces `^(@[a-z0-9-]+/)?[a-z0-9][a-z0-9-]*$`.

#### Manifest Error Types

[`crates/libaipm/src/manifest/error.rs:11-75`](https://github.com/TheLarkInn/aipm/blob/b034f7a3c3326ea746e8afd8bee63a7170899ca4/crates/libaipm/src/manifest/error.rs#L11-L75)

Eight variants: `Parse`, `MissingField`, `InvalidName`, `InvalidVersion`, `InvalidDependencyVersion`, `InvalidPluginType`, `ComponentNotFound`, `Io`, `Multiple`.

#### Security Verdict Pattern (Closest to Rule Severity)

[`crates/libaipm/src/linker/security.rs:9-71`](https://github.com/TheLarkInn/aipm/blob/b034f7a3c3326ea746e8afd8bee63a7170899ca4/crates/libaipm/src/linker/security.rs#L9-L71)

- `ScriptVerdict` enum: `Allowed` or `Blocked`
- `evaluate_scripts()` checks each script against an allowlist
- `has_blocked_scripts()` / `blocked_scripts()` query helpers
- This is the closest existing pattern to "rule produces verdict with severity" but only has two levels

#### Directory Walking with Ignore Patterns

[`crates/libaipm/src/migrate/discovery.rs:37-55`](https://github.com/TheLarkInn/aipm/blob/b034f7a3c3326ea746e8afd8bee63a7170899ca4/crates/libaipm/src/migrate/discovery.rs#L37-L55)

Uses `ignore` crate (workspace dep at `Cargo.toml:63`) with gitignore-aware traversal, explicit `.ai/` exclusion, optional `max_depth`.

#### CLI Subcommand Pattern (Clap)

[`crates/aipm/src/main.rs:12-136`](https://github.com/TheLarkInn/aipm/blob/b034f7a3c3326ea746e8afd8bee63a7170899ca4/crates/aipm/src/main.rs#L12-L136)

- `#[derive(Parser)]` on `Cli` struct with `#[command(subcommand)]`
- `#[derive(Subcommand)]` on `Commands` enum with variants: `Init`, `Install`, `Update`, `Link`, `Unlink`, `List`, `Migrate`
- `run()` matches on `cli.command` and dispatches to `cmd_*()` handler functions
- Errors: `writeln!(stderr, "error: {e}")` with `ExitCode::FAILURE`

[`crates/aipm-pack/src/main.rs:11-42`](https://github.com/TheLarkInn/aipm/blob/b034f7a3c3326ea746e8afd8bee63a7170899ca4/crates/aipm-pack/src/main.rs#L11-L42)

- Same pattern, currently only has `Init` variant
- The BDD feature specifies `aipm-pack lint` (not `aipm lint`), suggesting lint belongs in the pack CLI

#### Dry-Run Report Generation

[`crates/libaipm/src/migrate/dry_run.rs`](https://github.com/TheLarkInn/aipm/blob/b034f7a3c3326ea746e8afd8bee63a7170899ca4/crates/libaipm/src/migrate/dry_run.rs)

Generates markdown reports with sections grouped by artifact kind, summary tables with counts, per-artifact detail including source/target/files/conflicts. This is the closest existing pattern to a lint report output format.

#### Schema Enforcement via Serde

`Manifest` at [`types.rs:11`](https://github.com/TheLarkInn/aipm/blob/b034f7a3c3326ea746e8afd8bee63a7170899ca4/crates/libaipm/src/manifest/types.rs#L11) uses `#[serde(deny_unknown_fields)]` — TOML deserialization rejects unknown fields automatically.

### 5. What Does NOT Exist Yet

- No `lint` or `check` subcommand on either CLI
- No `validate` subcommand (validation runs internally during `load()`)
- No severity levels (error/warning/info) as a shared enum
- No rule IDs or diagnostic codes
- No shared diagnostic/report struct across modules
- No ignore-pattern system for suppressing lint findings
- No `--fix` auto-fix infrastructure
- No quality score computation

## Architecture Documentation

### Patterns to Reuse from Migrate

1. **Trait-based rule dispatch**: `Detector` trait → analogous `LintRule` trait
2. **Factory function per source type**: `claude_detectors()` → `claude_lint_rules()`, `copilot_lint_rules()`, etc.
3. **Source-type match dispatch**: `match source` at `mod.rs:264` → same pattern for lint adapter selection
4. **Filesystem abstraction**: `&dyn Fs` parameter enables mock-based testing
5. **Parallel execution**: `rayon::par_iter()` for running rules concurrently
6. **Error accumulation**: `validate.rs` `Vec<Error>` → `Vec<Diagnostic>` pattern

### Conventions Lint Rules Would Validate (from Issue #110)

1. **No plugin features inside `.claude` subfolders** — skills/commands/hooks/lsp/output styles must be in `.ai/` marketplace (supports ignore patterns)
2. **All skills/commands/agents must have `name` and `description` in YAML frontmatter** — currently parsed by detectors but not enforced as a requirement
3. **All markdown file references in a plugin must not be broken paths** — `${CLAUDE_SKILL_DIR}/scripts/` references and `references/` subdirectory links

### Conventions from BDD Feature (Additional Rules)

4. **SKILL.md must have `description` frontmatter field** (quality.feature:28)
5. **SKILL.md should not exceed 5000 tokens** (quality.feature:33)
6. **Agent definitions should have `tools` frontmatter** (quality.feature:38)
7. **Hook events must be valid event names** (quality.feature:43)

## Open Questions

1. **Which CLI gets the lint command?** The BDD feature specifies `aipm-pack lint`, but issue #110 says `aipm lint`. These could be separate commands or the issue title may be shorthand.

2. **Adapter scope for lint vs migrate**: Issue #110 says "use the same adaptor architecture that `aipm migrate` uses so that this can run against other ai agent tool folders". Does this mean lint should scan `.claude/`, `.github/copilot/`, `.opencode/` **as source formats**, or does it mean lint should validate `.ai/` marketplace plugins that were migrated from those sources?

3. **Ignore pattern format**: Issue #110 mentions "supports ignore patterns" for the rule checking plugin features aren't inside `.claude` subfolders. What format? Glob patterns? A `.lintignore` file? Inline comments?

4. **Frontmatter parser**: The detectors each implement their own line-by-line frontmatter parsing. Should a shared YAML frontmatter parser be extracted for lint rules to reuse, or should lint rules continue the existing pattern?

5. **Severity model**: The BDD feature uses both "warning" and "error" but the codebase has no formal severity enum. The `ScriptVerdict` pattern (Allowed/Blocked) and the `Error::Multiple` accumulation pattern exist as starting points.

6. **Rule ID system**: The BDD feature mentions "machine-readable error code" (line 19). No rule ID system exists yet. What format? (e.g., `AIPM001`, `skill/missing-description`, etc.)
