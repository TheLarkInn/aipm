# Research: The `edition` Field in AI Plugin Manifests — Purpose and Rationale

## Research Question

What is the purpose of the `Edition` field in AI plugin manifests generated by `aipm migrate`? Why does it exist, what values can it take, how is it used in the codebase, and what design rationale justifies its inclusion in an AI plugin manifest?

## Summary

The `edition` field in `aipm.toml` manifests is a **forward-looking schema reservation** borrowed from Rust's Cargo package manager. It currently has **no functional purpose** — no code reads, validates, or acts on its value. The field is hardcoded to `"2024"` across all five manifest generation paths and is defined as `Option<String>` with zero validation. It was not part of the original technical design (2026-03-09) but was added one week later when the init command was implemented, following the pattern documented in Cargo research. The design intent is to provide a **manifest versioning epoch** that will allow future aipm versions to change default behaviors (like dependency resolver algorithms or feature semantics) without breaking existing plugins — the same role `edition` plays in Cargo.

## Detailed Findings

### 1. What Is the Edition Field Today

The field is defined in the `Package` struct at [`crates/libaipm/src/manifest/types.rs:60-61`](https://github.com/TheLarkInn/aipm/blob/d399e8e9c061b520a5f25d9fd59639c754cfcf6c/crates/libaipm/src/manifest/types.rs#L60-L61):

```rust
/// Edition identifier.
pub edition: Option<String>,
```

Key characteristics:
- **Type**: `Option<String>` — optional, accepts any string value
- **Validation**: None. [`validate.rs`](https://github.com/TheLarkInn/aipm/blob/d399e8e9c061b520a5f25d9fd59639c754cfcf6c/crates/libaipm/src/manifest/validate.rs) contains zero references to `edition`
- **Default value**: Always hardcoded to `"2024"` in all generation paths
- **Runtime effect**: None. No code reads or branches on the edition value

### 3. The Cargo Inspiration

The `edition` field is modeled directly after Rust/Cargo's edition system, as documented in the project's own research at [`research/docs/2026-03-09-cargo-core-principles.md`](https://github.com/TheLarkInn/aipm/blob/d399e8e9c061b520a5f25d9fd59639c754cfcf6c/research/docs/2026-03-09-cargo-core-principles.md).

In Cargo, `edition` is part of the required trio (`name`, `version`, `edition`) in `[package]`. It serves a specific purpose:

- **Editions are opt-in behavior epochs**: Rust editions (2015, 2018, 2021, 2024) allow the language and toolchain to introduce breaking changes to defaults without breaking existing code. Each crate declares which edition it targets.
- **Resolver version selection**: The edition determines which dependency resolver is used by default:
  - Edition 2015/2018 → Resolver `"1"` (union of all features across the entire graph)
  - Edition 2021 → Resolver `"2"` (selective feature unification)
  - Edition 2024 → Resolver `"3"` (MSRV-aware resolution)
- **Backward compatibility without stagnation**: Old crates keep working under their declared edition. New crates get improved defaults. Different editions can coexist in the same dependency graph.

The Cargo research document (line 118) describes this:
> `[package]`: `name`, `version`, `edition` (required trio), plus `rust-version`, `description`, `license`, `publish`, `links`

### 4. Timeline: How Edition Entered the Manifest

The edition field was **not** part of the original aipm manifest design:

- **2026-03-09**: The foundational technical design spec ([`specs/2026-03-09-aipm-technical-design.md:340-373`](https://github.com/TheLarkInn/aipm/blob/d399e8e9c061b520a5f25d9fd59639c754cfcf6c/specs/2026-03-09-aipm-technical-design.md#L340-L373)) defines the plugin member manifest with `name`, `version`, `description`, `type`, `[dependencies]`, `[features]`, `[components]`, and `[environment]` — **no `edition` field**.
- **2026-03-09**: The Cargo core principles research documents how Cargo uses editions.
- **2026-03-16**: The init-workspace-marketplace spec ([`specs/2026-03-16-aipm-init-workspace-marketplace.md:297`](https://github.com/TheLarkInn/aipm/blob/d399e8e9c061b520a5f25d9fd59639c754cfcf6c/specs/2026-03-16-aipm-init-workspace-marketplace.md#L297)) shows `edition = "2024"` in the starter plugin manifest — **first appearance**.
- **2026-03-22 onward**: All subsequent specs include edition as part of the manifest boilerplate.
- **2026-03-24**: The suppress-plugin-manifest spec explicitly identifies edition as premature.

### 5. The Suppress-Manifest Spec's Commentary

The most direct acknowledgment of edition's current status comes from [`specs/2026-03-24-suppress-plugin-manifest-generation.md:34-36`](https://github.com/TheLarkInn/aipm/blob/d399e8e9c061b520a5f25d9fd59639c754cfcf6c/specs/2026-03-24-suppress-plugin-manifest-generation.md#L34-L36):

> **User confusion**: Running `aipm init` or `aipm migrate` produces `aipm.toml` files with `[dependencies]`, `version`, and `edition` fields that suggest a package management system. Users attempt to add dependencies, set versions, or publish — none of which work yet.

> **Premature commitment**: Generated manifests lock in schema decisions (hardcoded `version = "0.1.0"`, `edition = "2024"`) before the dependency resolver is built. When the resolver ships, these manifests may need updating.

> **No functional purpose**: Plugin-level manifests serve no runtime purpose today. Claude Code ignores them entirely.

### 7. Design Rationale: Why Include Edition in an AI Plugin Manifest

There is **no dedicated spec or design document** that explicitly justifies the edition field. Based on the evidence:

**The rationale is forward-looking schema evolution**, following Cargo's proven pattern:

1. **Future resolver behavior**: Just as Cargo ties resolver versions to editions, aipm could tie future dependency resolution strategies to edition values. A plugin declaring `edition = "2024"` would get one set of defaults; a future `edition = "2025"` could change how features unify, how version ranges resolve, or what validation rules apply.

2. **Non-breaking behavioral changes**: When aipm's dependency management system ships, the team may want to change defaults (e.g., stricter validation, different feature resolution). Edition lets them do this without breaking plugins that were created under the old assumptions — each plugin "locks in" its behavioral epoch.

3. **Schema versioning without a `schema_version` field**: Rather than a generic `schema_version`, the year-based edition communicates the same intent in a more user-friendly way that aligns with the Cargo ecosystem the project is modeled after.

4. **Signaling intent**: Even today, the field signals that `aipm.toml` is designed for a Cargo-like lifecycle (versioning, resolution, publishing) — not just a static config file.

## Architecture Documentation

### How Edition Fits in the Manifest Schema

```
aipm.toml (plugin member)
├── [package]
│   ├── name        ← required, validated (regex + semver)
│   ├── version     ← required, validated (semver)
│   ├── type        ← optional, validated (enum: skill|agent|mcp|hook|lsp|composite)
│   ├── edition     ← optional, NOT validated, no behavioral effect
│   ├── description ← optional, not validated
│   └── files       ← optional, not validated
├── [dependencies]  ...
├── [components]    ...
└── [environment]   ...
```

### The Cargo Pattern Being Followed

```
Cargo.toml [package]        →  aipm.toml [package]
├── name (required)          ├── name (required)
├── version (required)       ├── version (required)
├── edition (required)       ├── edition (optional, placeholder)
├── rust-version             ├── (not yet: aipm-version in [environment])
├── description              ├── description
└── ...                      └── type (aipm-specific)
```

In Cargo, `edition` unlocks specific resolver and language behaviors per epoch. In aipm, edition is reserved for the same pattern but has no implementation behind it yet.

## Open Questions

1. **What will edition control when the resolver ships?** Cargo ties resolver versions to editions. Will aipm follow the same pattern, or will edition govern different behaviors?
2. **Should edition become required?** In Cargo, edition is required. In aipm it is currently `Option<String>`. If it gains behavioral significance, it may need to become required with a default.
3. **What values should be valid?** Currently any string is accepted. Cargo uses year-based values (2015, 2018, 2021, 2024). Will aipm follow the same convention?
4. **Should edition be validated?** When the field gains meaning, validation logic will need to be added to `validate.rs` to reject unknown edition values.
