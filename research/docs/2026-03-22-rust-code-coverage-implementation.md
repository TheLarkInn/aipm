# Research: Strict Code Coverage as a Correctness Gate for Agentic Code Generation

## Research Question

How does Rust implement code coverage? How to collect unified coverage across unit tests, E2E tests (assert_cmd), and BDD tests (cucumber-rs) in this workspace? What are best practices from rust-lang/rust, Mozilla (grcov), and the broader ecosystem? What coverage thresholds and enforcement mechanisms exist? What are Windows-specific gotchas?

## Framing: Coverage as Correctness, Not Reporting

In this project, code is generated by LLMs (agentic code generation). This changes the purpose of coverage fundamentally:

- **Traditional framing**: Coverage is a *reporting metric* — "how much of our code is tested?" It's informational, used for trend tracking and team awareness.
- **This project's framing**: Coverage is a *correctness gate* — "did the LLM generate code that is actually reachable and exercised?" Uncovered branches may indicate hallucinated logic, dead code, or untested error paths that the LLM assumed it handled.

**Implications of this framing**:
1. **Branch coverage is mandatory**, not optional. Line coverage misses the most common LLM failure: generating the happy path test but skipping error branches. A single line with `if/else` shows 100% line coverage if either branch runs — branch coverage catches the untested side.
2. **90% branch coverage threshold, enforced immediately**. No ratcheting. If LLM-generated code can't hit 90%, it has structural problems that should block the PR.
3. **Coverage runs as a default step post-build**, not just in CI. The LLM should see coverage results before pushing.
4. **Nightly Rust for coverage is acceptable**. Branch coverage (`-C instrument-coverage=branch`) requires nightly. Since coverage is a correctness gate (not a compilation target), using nightly for the coverage job while keeping stable for build/test/clippy is sound. The coverage job validates test completeness, not compiler correctness.

## Summary

Rust's code coverage is built on LLVM's source-based instrumentation (`-C instrument-coverage`), stabilized since Rust 1.60. Branch coverage requires nightly (`-C instrument-coverage=branch`). The primary tool is **cargo-llvm-cov** (by taiki-e), which wraps the LLVM instrumentation pipeline and handles profraw merging automatically. For the aipm workspace, coverage can be collected across all three test tiers (92 unit tests, 33 E2E tests, BDD cucumber-rs scenarios) with a unified workflow. There is currently **zero coverage tooling** configured in this repository.

### Resolved Decisions

| Decision | Resolution | Rationale |
|----------|-----------|-----------|
| Branch coverage | **Mandatory** | LLMs generate happy-path tests; branch coverage catches untested error arms |
| Threshold | **90% immediately** | No ratcheting; LLM code that can't hit 90% has structural problems |
| Nightly for coverage | **Accepted** | Coverage is a correctness gate, not a compilation target |
| Doctest coverage | **Included** | Already on nightly; doctests are both documentation and correctness signals |
| External reporting | **Codecov + local --fail-under** | Hard gate via CLI flag; Codecov for PR annotations and trends |
| Node.js in CI | **Install it** | 5 scaffold E2E tests silently skip without it; unacceptable coverage gap |
| Post-build default | **Yes** | LLM should see coverage results before pushing |

## Detailed Findings

### 1. How Rust's Source-Based Coverage Works

**Source**: [Instrumentation-based Code Coverage -- The rustc Book](https://doc.rust-lang.org/beta/rustc/instrument-coverage.html)
**Source**: [LLVM Coverage Instrumentation -- Rust Compiler Dev Guide](https://rustc-dev-guide.rust-lang.org/llvm-coverage-instrumentation.html)

Rust uses LLVM's InstrProf framework for source-based code coverage:

1. **Compile with instrumentation**: `RUSTFLAGS="-C instrument-coverage"` injects counter increments at coverage-relevant points in the compiled binary. For branch coverage, `-C instrument-coverage=branch` inserts counters at every conditional branch point (`br i1` in LLVM IR).
2. **Run the binary**: At exit, the runtime writes a `.profraw` file containing raw counter data. The output path is controlled by `LLVM_PROFILE_FILE` env var.
3. **Merge profiles**: `llvm-profdata merge -sparse *.profraw -o coverage.profdata` combines multiple raw profiles.
4. **Generate reports**: `llvm-cov report/show/export` reads the merged profile + debug info from the binary to produce text, HTML, lcov, or JSON reports.

**LLVM_PROFILE_FILE pattern specifiers**:

| Pattern | Meaning |
|---------|---------|
| `%p` | Process ID (unique per process) |
| `%m` | Binary signature hash (unique per binary) |
| `%h` | Hostname |
| `%t` | TMPDIR |

Recommended pattern: `target/coverage/%p-%m.profraw` — guarantees unique files across processes and binaries.

**Why branch coverage requires nightly**: The `-C instrument-coverage=branch` flag is not yet stabilized. Basic line/region/function coverage was stabilized in Rust 1.60. Branch coverage inserts counters at every conditional branch point in the LLVM IR, and the mapping back to Rust source constructs (`match` arms, `?` operator, `&&`/`||` short-circuits, `let-else`, closures, async transforms) has edge cases still being refined. Tracking issue: [rust-lang/rust #79649](https://github.com/rust-lang/rust/issues/79649). The work is active, not stalled.

**What "nightly" means**: The nightly release channel of the Rust compiler, built from `main` every night. It includes experimental/unstable features behind feature gates. It is not "unstable" in the "crashes randomly" sense — it's the same compiler with gates unlocked. Using `cargo +nightly llvm-cov` for coverage while keeping `cargo +stable` for build/test/clippy is a well-established pattern.

### 2. Why Branch Coverage Matters for Agentic Code

**Line coverage** counts whether a line executed. **Branch coverage** counts whether both sides of every decision point were taken.

Example:
```rust
fn classify(score: u32) -> &'static str {
    if score >= 90 { "A" } else if score >= 70 { "B" } else { "C" }
}
```

A test with `classify(95)` gives **100% line coverage** (the whole expression executes). But branch coverage shows only 1 of 3 branches was taken. The `"B"` and `"C"` paths are completely untested.

LLMs tend to generate happy-path tests. They'll test that `parse_and_validate` works on valid input but skip the 4 different error branches. Branch coverage catches exactly this — "you tested that the `match` runs, but never hit the `Err` arms."

The workspace's strict lint configuration amplifies this: `unwrap_used`, `expect_used`, `panic`, `todo` are all denied, forcing LLM-generated code to handle error paths explicitly with `Result`/`?`. Branch coverage verifies those error paths are actually exercised by tests.

### 4. Tool Selection: cargo-llvm-cov

**Source**: [taiki-e/cargo-llvm-cov](https://github.com/taiki-e/cargo-llvm-cov)

The dominant modern tool. Wraps `rustc -C instrument-coverage` + `llvm-profdata` + `llvm-cov` into a single `cargo` subcommand.

- **Installation**: `cargo install cargo-llvm-cov` or `rustup component add llvm-tools-preview` + install binary
- **Workspace support**: `cargo llvm-cov --workspace` instruments and tests all crates
- **Output formats**: `--text`, `--html`, `--lcov`, `--json`, `--cobertura`
- **Branch coverage**: `--branch` flag (requires nightly)
- **Threshold enforcement**: `--fail-under-branches`, `--fail-under-lines`, `--fail-under-functions`, `--fail-under-regions`
- **Profile merging**: Automatic across `cargo test` invocations when using `--no-report` + `report`
- **Platform**: Works on Linux, macOS, and Windows
- **Maintenance**: Actively maintained by taiki-e (prolific Rust ecosystem contributor)

Key flags:

| Flag | Purpose |
|------|---------|
| `--workspace` | Test all workspace members |
| `--branch` | Enable branch coverage (nightly) |
| `--no-report` | Run tests and collect profraw without generating report |
| `report` | Generate report from previously collected profraw files |
| `--fail-under-branches <N>` | Fail if branch coverage < N% |
| `--lcov --output-path lcov.info` | Output lcov format for Codecov |
| `--html --output-dir coverage/` | Generate HTML report |
| `show-env` | Print env vars for external process instrumentation |
| `--ignore-filename-regex` | Exclude files from report (e.g., test code) |

**Rejected alternatives**:
- **grcov** (Mozilla): Multi-language aggregation tool. Overkill for pure Rust. cargo-llvm-cov is simpler and more widely adopted.
- **tarpaulin**: Linux x86_64 only (ptrace-based). Does NOT support Windows. Not viable for this repo's local dev workflow.

### 5. The Child Process Challenge (Critical for aipm)

Both E2E tests and BDD tests spawn `aipm` and `aipm-pack` as child processes via `assert_cmd::Command::cargo_bin()`. By default, **only the test binary is instrumented, not the spawned child binaries**.

#### The Problem

When `cargo llvm-cov` runs `cargo test`, it instruments and runs the test binaries. But `Command::cargo_bin("aipm")` spawns a separate process from the pre-built `target/debug/aipm` binary. If that binary wasn't built with `-C instrument-coverage`, its execution produces no coverage data.

#### Solution: cargo-llvm-cov's Workspace Instrumentation

`cargo llvm-cov --workspace` builds ALL workspace members (including binaries) with instrumentation. When `assert_cmd::Command::cargo_bin()` resolves the binary path, it finds the instrumented version in `target/`. The child process inherits `LLVM_PROFILE_FILE` from the environment (set by cargo-llvm-cov), so it writes its own `.profraw` file on exit.

**This works automatically** for binaries in the same workspace — no manual configuration needed. The key requirement is that `cargo llvm-cov --workspace` (not per-crate) is used, so all binaries are instrumented.

#### Fallback: Manual Environment Setup

If child process coverage is missing, use `cargo llvm-cov show-env` to get the env vars and manually set them:
```bash
eval $(cargo llvm-cov show-env --export-prefix)
cargo build --workspace  # builds instrumented binaries
cargo test --workspace   # runs tests, spawned binaries write profraw
cargo llvm-cov report --lcov --output-path lcov.info
```

### 6. BDD/Cucumber-rs Coverage Considerations

The BDD test harness at [`crates/libaipm/tests/bdd.rs`](https://github.com/TheLarkInn/aipm/blob/daf021333dc86c8bdff1913ced9dcbbfd121f75b/crates/libaipm/tests/bdd.rs) uses `harness = false`, meaning it has a custom `fn main()`. This is compatible with `cargo llvm-cov` because:

1. **The BDD binary itself is instrumented**: `cargo llvm-cov` instruments all test binaries, including those with `harness = false`. The `bdd` test binary runs as a single process, and its direct library calls (e.g., `libaipm::manifest::parse_and_validate` at line 255, `version::Requirement::select_best` at line 269) are fully covered.

2. **Spawned binaries via `run_command`**: The helper at bdd.rs:93-121 uses `Command::cargo_bin(binary)` to spawn `aipm` or `aipm-pack`. Same child-process considerations as E2E tests apply — use `--workspace` to ensure these binaries are instrumented.

3. **No special cucumber-rs configuration needed**: Coverage instrumentation is at the compiler level, transparent to the test framework.

### 8. Ecosystem Best Practices

#### rust-lang/rust

**Source**: [Rust Compiler Dev Guide — Coverage](https://rustc-dev-guide.rust-lang.org/llvm-coverage-instrumentation.html)

The rust-lang/rust repo does **not** enforce coverage thresholds on the compiler codebase. Their coverage CI jobs validate that the `-C instrument-coverage` feature produces correct output (coverage map correctness, end-to-end report validation), not that rustc itself has X% coverage.

#### Common Patterns from Popular Projects

**Source**: [taiki-e/cargo-llvm-cov CI](https://github.com/taiki-e/cargo-llvm-cov/blob/main/.github/workflows/ci.yml)
**Source**: [cargo-nextest coverage docs](https://nexte.st/docs/integrations/test-coverage/)

1. **Split lib from binary**: Maximize testable code by keeping binary crates thin. aipm already follows this pattern — `libaipm` contains all logic.

2. **Merge multiple test runs**: Use `--no-report` to accumulate profraw files across different test invocations, then `report` to generate a unified report:
   ```bash
   cargo llvm-cov clean --workspace
   cargo llvm-cov --no-report --workspace --branch   # unit + E2E + BDD
   cargo llvm-cov --no-report --doc                   # doctests
   cargo llvm-cov report --doctests --branch --lcov --output-path lcov.info
   ```

3. **Exclude test code from reports**: Use `--ignore-filename-regex 'tests/'` to avoid inflating numbers with test helper code.

#### Codecov Configuration

**Source**: [Codecov YAML Reference](https://docs.codecov.com/docs/codecovyml-reference)

```yaml
# codecov.yml
coverage:
  status:
    project:
      default:
        target: 90%      # absolute minimum — matches --fail-under-branches
        threshold: 1%    # allowed variance
    patch:
      default:
        target: 90%      # new code must be 90% branch-covered
  precision: 2
  round: down
ignore:
  - "tests/"
  - "crates/*/tests/"
  - "research/"
  - "specs/"
```

### 9. Windows-Specific Considerations

1. **LLVM tools availability**: `rustup component add llvm-tools-preview` works on Windows. cargo-llvm-cov handles tool path resolution.

2. **Path separators**: `LLVM_PROFILE_FILE` uses forward slashes even on Windows. cargo-llvm-cov handles this automatically.

3. **Symbol stripping**: The workspace's `[profile.release]` has `strip = "symbols"` — this would remove debug info needed for coverage. Coverage runs use the `dev`/`test` profile by default (not release), so this is not an issue for normal `cargo llvm-cov`.

4. **Long path issues**: Windows has a 260-character path limit. Deeply nested `target/` paths with profraw files could hit this. Mitigation: use short `LLVM_PROFILE_FILE` patterns.

5. **tarpaulin is not an option**: Linux x86_64 only. cargo-llvm-cov is the only viable cross-platform choice.

### 10. CI Workflow (Recommended)

```yaml
name: Coverage

on:
  push:
    branches: [main]
  pull_request:

env:
  CARGO_INCREMENTAL: 0
  CARGO_TERM_COLOR: always

jobs:
  coverage:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - uses: dtolnay/rust-toolchain@nightly
        with:
          components: llvm-tools-preview

      - uses: actions/setup-node@v4
        with:
          node-version: '22'

      - uses: taiki-e/install-action@cargo-llvm-cov

      - uses: Swatinem/rust-cache@v2

      # Merge coverage from all test types + doctests
      - name: Clean coverage artifacts
        run: cargo llvm-cov clean --workspace

      - name: Run all tests with branch coverage
        run: cargo llvm-cov --no-report --workspace --branch

      - name: Run doctests with branch coverage
        run: cargo llvm-cov --no-report --doc

      - name: Generate lcov report
        run: cargo llvm-cov report --doctests --branch --lcov --output-path lcov.info

      # Hard correctness gate
      - name: Enforce 90% branch coverage
        run: cargo llvm-cov report --doctests --branch --fail-under-branches 90

      - name: Upload to Codecov
        if: always()
        uses: codecov/codecov-action@v5
        with:
          files: lcov.info
          token: ${{ secrets.CODECOV_TOKEN }}
          fail_ci_if_error: true
```

## Architecture Documentation

The workspace follows the recommended pattern for maximizing coverage: thin binary crates (`aipm`, `aipm-pack`) that delegate to a library crate (`libaipm`). All 92 unit tests are in `libaipm`. E2E and BDD tests exercise the binaries as black boxes via `assert_cmd`. This architecture means most code is directly testable through unit tests (high branch coverage potential), while integration behavior is validated through E2E/BDD.

The strict lint configuration (`unwrap_used = "deny"`, `expect_used = "deny"`, `panic = "deny"`) forces all code paths to use proper error handling (`Result`/`?`), which naturally creates explicit branches that branch coverage can measure. This is a strong synergy: lints force explicit error paths, branch coverage verifies they're tested.
