# Research: Lint Rules for marketplace.json / plugin.json Validation

## Research Question

Solve GitHub issues #287, #288, #289, and #290 — four new lint rules for validating marketplace.json and plugin.json files in the aipm lint system.

## Summary

All four issues require new lint rules that operate on **JSON manifest files** (`marketplace.json` and `plugin.json`) rather than the frontmatter-based feature files (SKILL.md, agent .md, hooks.json) that existing rules target. The current discovery pipeline does NOT discover these JSON files as features, which means new scanning infrastructure is needed. The rules need to read and parse raw `serde_json::Value` objects (no typed Rust structs exist for these files) and cross-reference data between marketplace.json and plugin.json.

## Issue Breakdown

### Issue #290 — `marketplace/source-resolve` (Plugin source paths must resolve)
- For each plugin in `marketplace.json`, verify the `source` property exists
- Resolve the source path (relative to `.ai/`) and verify it exists on disk
- Default severity: **Error**

### Issue #289 — `marketplace/plugin-field-mismatch` (Name/description consistency)
- For each plugin in `marketplace.json`, load its corresponding `plugin.json`
- Verify `name` and `description` fields match between the two files
- Default severity: **Error** (fail by default, rangeable)
- Future: fixable (auto-sync)

### Issue #288 — `plugin/required-fields` (Required plugin.json fields)
- Each `plugin.json` must have non-empty: `name`, `description`, `version`
- Must also have `author` object with non-empty `name` and `email` subfields
- Default severity: **Error** (users can tune down)

### Issue #287 — `plugin/missing-registration` (Plugin registration completeness)
- Two sub-rules:
  1. Every plugin directory under `.ai/` must be registered in `marketplace.json`
  2. Every plugin directory must have a `.claude-plugin/plugin.json` file

## Detailed Findings

### Current Lint Architecture

#### Rule Trait
[`crates/libaipm/src/lint/rule.rs:16-50`](https://github.com/TheLarkInn/aipm/blob/9bc924a0266bd422f0c67e8b9ece82f23b487951/crates/libaipm/src/lint/rule.rs#L16-L50)

Every lint rule implements the `Rule` trait as a zero-sized unit struct:
- `id() -> &'static str` — hierarchical ID like `"plugin/required-fields"`
- `name() -> &'static str` — human-readable name
- `default_severity() -> Severity` — `Warning` or `Error`
- `help_url()` and `help_text()` — optional, defaults to `None`
- `check(source_dir, fs)` — directory-level scan returning `Vec<Diagnostic>`
- `check_file(file_path, fs)` — single-file check (default delegates to `check()` on parent)

#### Rule Registration
[`crates/libaipm/src/lint/rules/mod.rs:33-51`](https://github.com/TheLarkInn/aipm/blob/9bc924a0266bd422f0c67e8b9ece82f23b487951/crates/libaipm/src/lint/rules/mod.rs#L33-L51)

Rules are dispatched by `FeatureKind` via `quality_rules_for_kind()`:
- `FeatureKind::Skill` → 8 rules
- `FeatureKind::Agent` → 1 rule
- `FeatureKind::Hook` → 2 rules
- `FeatureKind::Plugin` → 1 rule (`BrokenPaths` only)

**To add new rules**: create a rule file in `crates/libaipm/src/lint/rules/`, add `pub mod` in `mod.rs`, and add `Box::new(...)` to the appropriate `FeatureKind` arm.

#### Discovery Pipeline
[`crates/libaipm/src/discovery.rs`](https://github.com/TheLarkInn/aipm/blob/9bc924a0266bd422f0c67e8b9ece82f23b487951/crates/libaipm/src/discovery.rs)

The lint pipeline discovers features via `discover_features()`. Currently `FeatureKind::Plugin` is only triggered for `aipm.toml` at exactly `.ai/<plugin>/aipm.toml` (line ~296-303). Neither `marketplace.json` nor `plugin.json` are discovered as features.

#### Lint Pipeline Flow
[`crates/libaipm/src/lint/mod.rs:115-162`](https://github.com/TheLarkInn/aipm/blob/9bc924a0266bd422f0c67e8b9ece82f23b487951/crates/libaipm/src/lint/mod.rs#L115-L162)

1. `discover_features()` walks the project tree
2. Optional `--source` filter applied
3. For each feature: `run_rules_for_feature()` → `quality_rules_for_kind()` → `rule.check_file()`
4. Sort diagnostics, count errors/warnings, return `Outcome`

### Source Path Resolution

Source paths in marketplace.json are relative paths formatted as `"./<name>"` (e.g., `"./starter-aipm-plugin"`). These are relative to the `.ai/` directory. To resolve: strip the leading `./`, join with the `.ai/` directory path.

### Scanning Infrastructure (scan.rs)
[`crates/libaipm/src/lint/rules/scan.rs`](https://github.com/TheLarkInn/aipm/blob/9bc924a0266bd422f0c67e8b9ece82f23b487951/crates/libaipm/src/lint/rules/scan.rs)

Currently provides `scan_skills()`, `scan_agents()`, `scan_hook_files()` for directory-level iteration, and `read_skill()`, `read_agent()`, `read_hook()` for single-file reads. **No marketplace.json or plugin.json scanning functions exist yet.**

New functions needed:
- `scan_marketplace_json(ai_dir, fs)` — reads `.ai/.claude-plugin/marketplace.json`, returns parsed JSON
- `scan_plugin_json(plugin_dir, fs)` — reads `.ai/<plugin>/.claude-plugin/plugin.json`, returns parsed JSON
- `scan_plugin_dirs(ai_dir, fs)` — lists plugin directories under `.ai/` (excluding `.claude-plugin`)

### Existing Rule Patterns to Follow

#### Pattern: Field validation (for #288)
Model after [`skill_missing_name.rs`](https://github.com/TheLarkInn/aipm/blob/9bc924a0266bd422f0c67e8b9ece82f23b487951/crates/libaipm/src/lint/rules/skill_missing_name.rs) — check if required fields exist and are non-empty. Produce one diagnostic per missing field.

#### Pattern: Path resolution (for #290)
Model after [`broken_paths.rs`](https://github.com/TheLarkInn/aipm/blob/9bc924a0266bd422f0c67e8b9ece82f23b487951/crates/libaipm/src/lint/rules/broken_paths.rs) — resolve relative paths and check `fs.exists()`.

## Architecture: Design Considerations

### Key Challenge: Discovery Gap

The current `discover_features()` pipeline does NOT discover `marketplace.json` or `plugin.json` files. The new rules need these files. Two approaches:

**Option A: Extend discovery** — Add `FeatureKind::Marketplace` and `FeatureKind::PluginManifest` to the discovery pipeline. This integrates cleanly but requires changes to `discovery.rs`.

**Option B: Marketplace-level rules** — New rules use `check()` with the `.ai/` directory, scanning for marketplace.json and plugin.json themselves (similar to how existing rules use `scan_skills()`). This requires a new dispatch mechanism since current rules are feature-level, not marketplace-level.

**Option C (simplest): Attach to existing FeatureKind::Plugin** — When `aipm.toml` is discovered at `.ai/<plugin>/aipm.toml`, the new rules run against the plugin directory and also read marketplace.json. However, this fails if no `aipm.toml` exists (which is the common case for init'd workspaces — they generate plugin.json but not aipm.toml).

**Recommended approach**: Extend discovery to recognize `marketplace.json` (at `.ai/.claude-plugin/marketplace.json`) and `plugin.json` (at `.ai/<plugin>/.claude-plugin/plugin.json`) as new feature kinds. This is the most natural integration and avoids special-casing.

### Proposed New FeatureKind Variants

```
FeatureKind::Marketplace  — .ai/.claude-plugin/marketplace.json
FeatureKind::PluginJson   — .ai/<plugin>/.claude-plugin/plugin.json
```

## Open Questions

1. **Discovery approach**: Should `marketplace.json` and `plugin.json` be discovered as new `FeatureKind` variants, or should rules scan independently? Extending discovery is cleaner but touches more files.

2. **Author field in #288**: Current plugin.json files do NOT have an `author` field. The `workspace_init` and `migrate` generators would need updating to include `author: { name, email }` to avoid all newly-created plugins immediately failing this lint. Should the generators be updated as part of this work, or should the rule default to `Warning` initially?

3. **#287 scope**: The issue mentions two sub-rules — should they be one rule producing multiple diagnostics, or two separate rule IDs? Two separate IDs give users more granular config control.

4. **Fixture `.claude-plugin/` directory naming**: The on-disk path uses `.claude-plugin/` (not `.claude-plugins/`). The issue body references `.claude-plugins/` — the code uses `.claude-plugin/` (singular). The rules should use the code's convention.

5. **Which marketplace.json directories to check**: Only `.ai/.claude-plugin/marketplace.json`, or also `.claude/.claude-plugin/marketplace.json` if it exists? Current generation only creates under `.ai/`.
