# Research: scaffold-plugin.ts Missing Autoenable + Marketplace.json Registration

## Research Question

The starter plugin's `scaffold-plugin.ts` TypeScript script (generated by `generate_scaffold_script()` in Rust) creates new plugins under `.ai/` but does NOT:
1. Auto-enable the new plugin in `.claude/settings.json` `enabledPlugins`
2. Register the new plugin in `.ai/.claude-plugin/marketplace.json`

What changes are needed, and what test coverage must be added?

## Summary

The `scaffold-plugin.ts` script is embedded as a Rust string literal in `generate_scaffold_script()` at `crates/libaipm/src/workspace_init/mod.rs:285-325`. When a user invokes it via `node --experimental-strip-types .ai/starter-aipm-plugin/scripts/scaffold-plugin.ts <plugin-name>`, it creates the plugin directory tree, `aipm.toml`, `SKILL.md`, and `plugin.json` — but stops there. It does not update the two registration files that make the plugin discoverable and enabled by Claude Code.

The Rust-side `aipm init` command correctly handles both registration tasks for the **starter** plugin (via `generate_marketplace_json()` and `adaptors/claude.rs`), so the patterns and JSON structures already exist. The TypeScript script simply needs to replicate them for **newly scaffolded** plugins.

## Detailed Findings

### 1. Current scaffold-plugin.ts Behavior

The script (`mod.rs:285-325`) performs these steps:

1. **Parse CLI arg** — reads `process.argv[2]` as plugin name
2. **Resolve paths** — `aiDir = <cwd>/.ai`, `pluginDir = <cwd>/.ai/<name>`
3. **Guard** — exits with error if `pluginDir` already exists
4. **Create directories** — `.claude-plugin/`, `skills/<name>/`, `agents/`, `hooks/`
5. **Write `aipm.toml`** — `[package]` + `[components]` with skills only
6. **Write `SKILL.md`** — starter skill with YAML frontmatter
7. **Write `plugin.json`** — `{ name, version, description }`
8. **Print success** — `Created .ai/<name>/ with starter structure`

**Files created for a plugin named `my-plugin`:**

| File | Content |
|------|---------|
| `.ai/my-plugin/.claude-plugin/plugin.json` | JSON plugin descriptor |
| `.ai/my-plugin/aipm.toml` | TOML package manifest |
| `.ai/my-plugin/skills/my-plugin/SKILL.md` | Starter skill |
| `.ai/my-plugin/agents/` | Empty directory |
| `.ai/my-plugin/hooks/` | Empty directory |

### 2. Missing: marketplace.json Registration

**File location:** `.ai/.claude-plugin/marketplace.json`

**Current structure** (created by `generate_marketplace_json()` at `mod.rs:371-403`):

```json
{
  "name": "local-repo-plugins",
  "owner": {
    "name": "local"
  },
  "metadata": {
    "description": "Local plugins for this repository"
  },
  "plugins": [
    {
      "name": "starter-aipm-plugin",
      "source": "./starter-aipm-plugin",
      "description": "Default starter plugin — scaffold new plugins, scan your marketplace, and log tool usage"
    }
  ]
}
```

**What the script must do:** Read `.ai/.claude-plugin/marketplace.json`, parse it, append a new entry to the `plugins` array:

```json
{
  "name": "<plugin-name>",
  "source": "./<plugin-name>",
  "description": "TODO: describe <plugin-name>"
}
```

Then write it back with `JSON.stringify(data, null, 2)`.

**Conventions observed in existing code:**
- `source` is the directory name prefixed with `./` (relative to `.ai/`)
- `description` matches the `aipm.toml` package description
- The marketplace name `"local-repo-plugins"` is the top-level `name` field

### 3. Missing: settings.json Autoenable

**File location:** `.claude/settings.json`

**Current structure** (created by `adaptors/claude.rs:28-42`):

```json
{
  "extraKnownMarketplaces": {
    "local-repo-plugins": {
      "source": {
        "source": "directory",
        "path": "./.ai"
      }
    }
  },
  "enabledPlugins": {
    "starter-aipm-plugin@local-repo-plugins": true
  }
}
```

**What the script must do:** Read `.claude/settings.json`, parse it, add a new key to `enabledPlugins`:

```json
"<plugin-name>@local-repo-plugins": true
```

The key format is `<plugin-name>@<marketplace-name>` where `<marketplace-name>` is always `"local-repo-plugins"` (from `marketplace.json` top-level `name`).

**Reference code in Rust adaptor (`claude.rs:92-97`):**
```rust
let enabled = obj.entry("enabledPlugins").or_insert_with(|| serde_json::json!({}));
if let Some(enabled_obj) = enabled.as_object_mut() {
    enabled_obj
        .entry("starter-aipm-plugin@local-repo-plugins")
        .or_insert(serde_json::json!(true));
}
```

The TypeScript equivalent should:
1. Read the file
2. Parse as JSON
3. Ensure `enabledPlugins` object exists (create if missing)
4. Add `<plugin-name>@local-repo-plugins: true` if not already present
5. Write back with pretty-printing

### 4. Updated scaffold-plugin.ts Requirements

After the fix, the script should:

1. Create plugin directory structure (existing behavior)
2. Write `aipm.toml`, `SKILL.md`, `plugin.json` (existing behavior)
3. **NEW**: Read and update `.ai/.claude-plugin/marketplace.json` — append plugin entry to `plugins` array
4. **NEW**: Read and update `.claude/settings.json` — add `<name>@local-repo-plugins: true` to `enabledPlugins`
5. Print success message (existing behavior)

Edge cases to handle:
- `marketplace.json` does not exist (create it with default structure)
- `settings.json` does not exist (create it with default structure including `enabledPlugins`)
- `enabledPlugins` key does not exist in `settings.json` (create it)
- Plugin already exists in `marketplace.json` or `settings.json` (skip, don't duplicate)

### 5. Impact on Rust Code

The change is in `generate_scaffold_script()` at `mod.rs:285-325`. The Rust function returns a string containing the TypeScript source. The string must be updated to include the new marketplace.json and settings.json update logic.

No other Rust functions need to change — the marketplace.json structure, settings.json structure, and all other scaffolding logic remain the same.

## Architecture Documentation

### Pattern: Embedded TypeScript in Rust String Literals

The project uses a pattern where TypeScript source code is embedded as Rust string literals in `generate_*()` functions. The Rust build system writes these strings to disk during `scaffold_marketplace()`. This means:

- TypeScript changes require modifying the Rust source
- TypeScript is not independently testable without first running `aipm init`
- Node.js built-ins only (no npm dependencies) — uses `fs`, `path`, `process`
- Requires Node.js >= 22.6.0 with `--experimental-strip-types`

### Pattern: Plugin Registration is Two-Step

For a plugin to be fully discoverable and enabled by Claude Code, two files must be updated:
1. **Discovery**: `.ai/.claude-plugin/marketplace.json` `plugins` array
2. **Activation**: `.claude/settings.json` `enabledPlugins` object

The Rust `aipm init` command handles both for the starter plugin. The TypeScript scaffold script must handle both for new plugins.

## Open Questions

1. **Should the scaffold script handle the case where `.claude/settings.json` doesn't exist at all?** If `aipm init` was run, it will exist. But if someone manually created `.ai/`, it might not.
2. **Should the marketplace name `"local-repo-plugins"` be read from the existing `marketplace.json` rather than hardcoded?** This would be more robust but adds complexity.
3. **Error handling in TypeScript**: The current script uses `process.exit(1)` for errors. Should the new file-reading/writing logic follow the same pattern, or should it warn and continue?
4. **Should the spec (`specs/2026-03-20-better-default-plugin.md`) be updated** to document these new requirements before implementation?
