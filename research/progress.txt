## 2026-05-01 — Feature 2: Engine-root inference helper — DONE

Added `crates/libaipm/src/discovery/source.rs` with `pub fn infer_engine_root(path: &Path, project_root: &Path) -> Option<(Engine, PathBuf)>`.

Behavior:
- Walks path.ancestors() upward until project_root is reached.
- Returns the FIRST ancestor whose file_name matches `.claude` (Claude), `.github` (Copilot), or `.ai` (Ai). Innermost wins, mirroring the unified-discovery design where each feature is associated with the engine that authored it, not the marketplace host that contains it.
- Handles non-UTF-8 path components via `to_string_lossy()` — invalid UTF-8 simply doesn't match any engine root.
- Returns None if path == project_root, or if no engine ancestor exists.

Re-exported from `discovery/mod.rs` as `pub use source::infer_engine_root`.

Tests added: 14 covering .claude canonical, .github canonical, .github/copilot/skills/<x> (issue #725 layout), .ai/<plugin>, nested .ai/<plugin>/.claude (innermost wins), no-engine-ancestor, path == project_root, project_root is engine dir, relative paths with and without dot prefix, deeply nested .ai > .github > .claude (innermost wins), unrelated dotted dirs ignored, path outside project_root, project_root at filesystem root.

## 2026-05-01 — Feature 3: Skill layout grammar (THE customer fix for #725) — DONE

Added `crates/libaipm/src/discovery/layout.rs` with `pub fn match_skill(path, engine, source_root) -> Option<DiscoveredFeature>` plus a private `pick_layout_for_skill` helper.

Behavior:
- Case A: any ancestor name between path's parent and source_root (exclusive) literally equals "skills" → match as Skill. This is the customer fix for #725 (`.github/copilot/skills/<x>/SKILL.md` works because "skills" is in the ancestors list).
- Case B: engine == Copilot AND grandparent (ancestors[1]) == "copilot" → match as Layout::CopilotSubroot (existing aipm accommodation `.github/copilot/<name>/SKILL.md`).
- pick_layout_for_skill pattern-matches the ancestors slice to choose the right Layout variant. Arms ordered most-specific-first:
  1. `[_, "skills", "copilot", ..]` → CopilotSubrootWithSkills (#725)
  2. `[_, "copilot", ..]` → CopilotSubroot
  3. `[_, "skills", "<.dot>", _, ..]` → AiNested
  4. `[_, "skills", _, ..]` if engine==Ai → AiPlugin
  5. default → Canonical

Re-exported as `pub use layout::match_skill` from `discovery/mod.rs`.

Tests (17 total): each of 5 layouts (canonical Claude, canonical Copilot, CopilotSubroot, CopilotSubrootWithSkills with #725's exact path, AiPlugin, AiNested), the all-three-skills #725 batch, flat skills layout, structural fields (feature_dir + path), plus 5 negative cases (skill at engine root, skill directly under copilot dir without a <name> sub-dir, copilot subroot only matches Copilot engine, skill with no skills ancestor, AiPlugin gated to Ai engine).

## 2026-05-01 — Feature 4: Layout matchers for non-skill features — DONE

Added five non-skill matchers to `crates/libaipm/src/discovery/layout.rs`:

- `match_agent(path, engine, source_root)` — `*.md` whose parent directory is named `agents`. Mirrors today's `classify_feature_kind` agent branch (`discovery_legacy.rs:265-266`). Instruction-file precedence is the dispatcher's responsibility (later spec feature).
- `match_hook(path, engine, source_root)` — `hooks.json` whose parent is either the engine source root (CopilotHookDetector accommodation) OR a directory literally named `hooks` (today's classify_feature_kind line 267-268).
- `match_plugin(path, engine, source_root)` — `aipm.toml` with grandparent `.ai`. Mirrors today's line 269-270.
- `match_marketplace(path, engine, source_root)` — `marketplace.json` with parent `.claude-plugin` and grandparent `.ai`. Mirrors lines 271-275.
- `match_plugin_json(path, engine, source_root)` — `plugin.json` with parent `.claude-plugin` and great-grandparent `.ai`. Mirrors lines 276-287.

All non-skill matchers return `Layout::Canonical` since skill is the only feature kind whose layout shape varies. `feature_dir` is set to `path.parent()` per spec.
